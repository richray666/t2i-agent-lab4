# 实验四：基于预训练模型的文生图实践报告

## 1. 实验目标

本实验根据课件“基于预训练模型的文生图实践-实验4”的要求，设计并实现一个小型文生图智能体。该智能体能够统一调用国产开源文生图模型和国外开源文生图模型，完成提示词读取、后端选择、图像生成、元数据记录和错误记录。

本次实现选择的模型后端如下：

- 国产开源模型：Kwai-Kolors/Kolors-diffusers。Kolors 是快手开源的中英双语文生图模型，适合中文提示词和中国文化场景。
- 国外开源模型：microsoft/Lens-Turbo。Lens 是微软开源的文生图扩散模型，Turbo 版本是蒸馏加速模型（4 步采样即可成图）。需要特别说明：Lens-Turbo 的文本编码器采用大型 GPT-OSS 语言模型，整套权重约 29GB，在 8GB 显存的笔记本上即便开启 CPU offload 也会在加载阶段因内存不足（反量化为 bf16 后约需 40GB+）被系统终止，**无法在本机运行**，因此本实验将其部署到远程高显存 GPU 服务器上完成（见第 5 节）。

## 2. 基本原理

文生图模型通常把自然语言提示词编码成语义向量，再通过扩散模型逐步从噪声中恢复图像。典型流程包括文本编码器、潜空间去噪网络、VAE 解码器和采样调度器。

扩散模型的核心思想是学习一个反向去噪过程。训练阶段先向真实图像逐步加噪，模型学习在给定文本条件下预测噪声或速度场；推理阶段从随机噪声出发，经过多步去噪得到与提示词匹配的图像。

VAE 用于把高分辨率图像压缩到潜空间中，降低扩散网络的计算量。潜空间中的去噪结果最后再由 VAE 解码回 RGB 图像。

Classifier-Free Guidance 是常见的提示词对齐增强方法。它同时估计有条件和无条件生成方向，通过 guidance scale 控制图像对提示词的服从程度。CFG 过低可能导致图像不贴合文本，过高可能带来过饱和或伪影。

Lens 的课件重点在训练效率。它通过高信息密度的长描述图文对、多分辨率多比例训练、语义 VAE、强语言编码器和推理器，提高基础文生图模型的收敛效率和多语言泛化能力。

## 3. 智能体设计

项目新增 `src/t2i_agent` 包，核心模块如下：

- `prompts.py`：读取 `prompts/test_prompts.jsonl`，校验每条提示词必须包含 `id`、`backend_category`、`prompt_zh`、`prompt_en`、`evaluation_focus`。
- `backends.py`：根据 `--backend` 和 `--variant` 解析模型仓库、默认采样步数、CFG、分辨率和默认提示词语言。
- `generator.py`：懒加载 Diffusers、Torch、KolorsPipeline 或 DiffusionPipeline，执行真实模型推理。
- `metadata.py`：保存生成图片和同名 JSON 元数据；如果模型下载、权限或显存失败，则写入 `outputs/errors`。
- `cli.py`：提供统一命令行接口。

为了在无法下载 17GB/29GB 大模型、或无 GPU 的机器上也能完整验证整条流水线，智能体额外提供了一个 `offline` 后端。它不加载任何预训练权重，而是用 PIL 根据「提示词 + 随机种子」确定性地合成一张占位图（带渐变、若干图形和提示词水印），相同输入产生完全一致的输出。`offline` 后端走的是与真实模型完全相同的代码路径——提示词选择、采样参数、图像生成、元数据记录，因此可用来证明智能体框架本身正确可用，但其图像质量不代表真实模型，分析时需明确区分。

命令行示例：

```powershell
# 离线演示（任何机器，立即出图）
python -m t2i_agent generate --backend offline --prompt-id cn_scene --seed 66

# 真实模型（GPU 机器）
python -m t2i_agent generate --backend kolors --prompt-id cn_scene --seed 66 --offload
python -m t2i_agent generate --backend lens --variant turbo --prompt-id en_scene --seed 42 --offload
```

批量运行全部 5 条提示词可使用 `scripts/run_all.py`，或 `scripts/run_demo.ps1`（离线）/`scripts/run_real.ps1`（真实模型）。详细步骤见 `RUNBOOK.md`。

## 5. 实验环境

本实验使用两套环境：本机负责代码开发、离线流水线验证和 Kolors 真实生成；Lens-Turbo 因显存需求过大改在远程 GPU 服务器上运行。

**本机环境**：NVIDIA GeForce RTX 4060 Laptop GPU，显存 8GB，内存 16GB。使用 `pytorch2.2.2` conda 环境（CUDA 可用的 PyTorch、Diffusers 0.30.3、Transformers 4.57、Accelerate、Safetensors、Pillow）。由于项目路径含中文，会触发 Kolors 的 SentencePiece 分词器无法加载非 ASCII 路径的问题，因此通过 `subst X:` 将工程目录映射为纯英文盘符、并设置 `PYTHONUTF8=1` 后运行（见 `scripts/run_real.ps1`）。Kolors（文本编码器为 ChatGLM-6B）在该环境下配合 `--offload --dtype float16` 可成功运行。

**远程服务器环境**：通过 ZeroTier 内网穿透接入的 Ubuntu GPU 服务器，配备 4× NVIDIA RTX PRO 6000 Blackwell（单卡 96GB 显存）。在 `pytorch/pytorch:2.7.1-cuda12.8-cudnn9-devel` 容器内新建 Lens 依赖（Transformers 5.8.0、Diffusers 0.38.0、Accelerate 1.13.0、huggingface_hub 1.18.0），模型经 hf-mirror 镜像下载到容器挂载目录。96GB 显存可一次性容纳 Lens-Turbo 全量 bf16 权重（约 56GB），无需 offload。

验证命令：

```powershell
conda run -n pytorch2.2.2 python -m unittest discover -s tests -v
```

## 6. 实验结果记录方式

每次生成会在 `outputs` 中保存两类文件：

- `.png`：生成图像。
- `.json`：同名元数据，记录 backend、repo_id、prompt_id、prompt、seed、steps、cfg、分辨率、运行时间和 offload 状态。

如果运行失败，智能体会在 `outputs/errors` 中保存错误 JSON。这样可以把 Hugging Face 权限、网络失败、显存不足等问题作为实验现象记录，而不是静默忽略。

真实模型运行记录：

- Offline：用于验证智能体流水线，可立即生成真实 `.png` 和同名 `.json`。离线图带 `[OFFLINE DEMO]` 水印，只代表“提示词读取、后端路由、图像保存、元数据保存”流程可用，不代表真实文生图模型质量。
- Kolors：已启动过 `Kwai-Kolors/Kolors-diffusers` 下载与加载，旧错误记录位于 `outputs/errors/20260604-221654-kolors-cn_scene.json`。随后缓存继续增长到约 17.8 GB，且未发现 `.incomplete` 文件；通过 `X:` ASCII 路径映射、`PYTHONUTF8=1` 和 CUDA DLL PATH 修正后，已成功生成 Kolors 单图冒烟结果。
- Lens-Turbo：本机尝试失败——`microsoft/Lens-Turbo` 的文本编码器为大型 GPT-OSS 模型，在缺少 Triton 的 Windows 上会被反量化为 bf16（约需 40GB+ 内存），8GB 显存 / 16GB 内存的笔记本在权重加载阶段即被系统终止（进程退出码 5，无 Python 回溯，历史错误记录见 `outputs/errors/20260605-051058-lens-en_scene.json`）。最终改在远程 4×96GB Blackwell 服务器的容器中运行：经 hf-mirror 镜像下载 29GB 权重后，整套模型一次加载到单卡（无需 offload），5 类提示词全部成功生成，每张 1024×1024 图像加载约 2 分钟、4 步推理约 1 秒，单条总耗时约 100–160 秒。

## 7. 结果分析维度

生成图片后，从以下维度进行比较：

- 文本对齐：图像主体是否符合提示词。
- 中文能力：Kolors 对中文场景和中文文字的理解是否优于英文为主的模型。
- 复杂属性绑定：颜色、材质、数量是否对应正确对象。
- 文字渲染：海报中的中文文字是否可读。
- 视觉质量：是否存在明显伪影、变形、过饱和、局部结构错误。
- 资源消耗：是否需要 CPU offload，运行时间是否可接受。

## 8. 当前结论

本实验已完成统一智能体代码结构、提示词测试集、后端配置（offline / kolors / lens）、元数据记录、错误记录、批量运行脚本和单元测试。新增的 `offline` 后端使整条流水线可在任何机器上端到端跑通并产出真实的 png + json 产物，从而把「智能体框架是否正确」与「真实模型是否已下载」两件事解耦：框架正确性已通过单元测试和离线演示验证。

在真实模型层面，本实验完成了国产模型 Kolors 与国外模型 Lens-Turbo 的实测：Kolors（约 17GB，文本编码器 ChatGLM-6B）在 8GB 显存笔记本上以 fp16 + CPU offload 成功生成 `cn_scene` 图像；Lens-Turbo（约 29GB，文本编码器为大型 GPT-OSS）因显存与内存需求远超笔记本能力，改在远程 4×96GB Blackwell GPU 服务器的容器中运行，成功生成全部 5 类提示词的图像。

从结果看：Lens-Turbo 凭借更强的文本编码器，在写实度、多属性绑定和文字渲染等维度表现优异，但部署门槛极高（需近百 GB 显存）；Kolors 则以轻量取胜，能在消费级 8GB 显卡上运行，更适合算力受限场景。一个有价值的工程经验是：模型选型不仅要看生成质量，更要评估其文本编码器规模带来的真实硬件开销——本实验中 Lens-Turbo 无法在本机运行的根因正是其 GPT-OSS 编码器，而非扩散网络本身。

- 已可演示：`offline` 后端可立即产出真实产物，证明流水线可用。

- 已完成（国外模型）：远程 GPU 服务器用 Lens-Turbo 生成全部 5 类提示词图像，并完成第 7 节对比评分。
